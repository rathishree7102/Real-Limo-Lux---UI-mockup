Detailed flow

Admin journey steps

How an operations admin moves through the actual portal screens in Admin.html, from secure access to the four core operating modules and Settings.

1
Admin lands on login. The login screen collects email or username, password, remember-me preference, and exposes the forgot-password path.Routes: #admin-login, #admin-forgot, #admin-reset.
2
Recovery branch verifies admin access. Forgot password sends an OTP to the admin email; reset requires OTP, new password, confirm password, and password rules.
3
Successful login opens Booking Management. The portal shows the sidebar, topbar search, profile menu, and default Bookings workspace.Visible sidebar: Bookings, Driver Management, Fleet Management, Customers, Settings.
4
Admin handles Booking Management. New Requests, Bookings, and Add Booking tabs support request review, booking lookup, booking detail panels, confirmation, cancellation, resource reassignment, and manual booking creation.
5
Admin handles Driver Management. The Drivers tab supports search, availability and license filters, driver detail panel, driver history, leave, suspension, deletion, and the Add Driver form with document upload.
6
Admin handles Fleet Management. The Fleet tab supports vehicle search and filters, status/action panel, vehicle details, vehicle history, maintenance/disable/delete flows, and the Add Fleet form with photos and compliance documents.
7
Admin handles Customers. Client Profiles supports VIP filtering, customer search, customer rows with ID/contact details, customer detail access, and VIP/Standard updates.
8
Admin configures Settings. Settings exposes Communication Settings in the active tab, including notification rules, reminder timing, and the Test Notification panel; the HTML also contains a hidden General Settings panel.
9
Admin logs out from profile. Profile menu shows Operations Admin and returns the user to the login screen.

Core workflow

Booking module lifecycle

The booking lifecycle follows the three tabs and embedded panels present in Admin.html: New Requests, Bookings, and Add Booking. It focuses on request review, confirmation/cancellation, viewing assigned bookings, reassignment modals, and manual booking creation.

1. Intake

New customer ride requests arrive in booking-requests. Admin filters by pickup date and opens the detail side panel to review customer, route, schedule, duration, and requested vehicle preference.

2. Review

Admin checks resource availability, customer notes, route/stops, and VIP context. Request can be confirmed or cancelled; cancellation should capture a reason.

3. Booking creation

Confirmed requests move into the booking list. Staff can also create manual bookings from the Add Booking tab with customer, route, stops, vehicle type, date, time, and duration.

4. Assignment

Booking detail panels show vehicle and driver assignments. Reassignment modals support changing the assigned vehicle or chauffeur.

5. Status-aware actions

Confirmed bookings expose reassignment and cancellation actions. Completed and cancelled bookings are still viewable, but active controls are hidden in the panel.

6. Closure

Completed and cancelled bookings remain in the booking list for reference. Cancellation uses a modal and should preserve the reason in implementation.

User stories

Admin-facing user stories grouped by the screens and panels in the supplied Admin.html flow.

Authentication & Shell

AD-01 - Secure login

As an admin, I want to log in securely so that only authorized staff can access operations.

Acceptance criteria
  • Login accepts email/username and password.
  • Successful login opens the Bookings workspace.
  • Unauthenticated users see no admin sidebar/topbar.
AD-02 - Password reset

As an admin, I want to reset my password with OTP verification so that I can recover account access safely.

Acceptance criteria
  • Forgot Password collects admin email.
  • Reset requires OTP, new password, and confirmation.
  • Password guidance is visible before submit.
AD-03 - Global search

As an operations admin, I want global search for booking/customer/contact data so that I can jump to records quickly.

Acceptance criteria
  • Search placeholder covers booking ID, customer, phone, and email.
  • Results should focus on records available in the active admin workspace.
AD-04 - Profile logout

As an admin, I want profile access and logout so that I can end my session from any admin screen.

Acceptance criteria
  • Profile menu displays admin identity.
  • Logout returns to the login route.

Booking Operations

AD-05 - Review new requests

As an operations admin, I want a New Requests queue so that I can review incoming ride requests before accepting them.

Acceptance criteria
  • Queue lists request ID, customer, pickup, drop-off, date/time, duration, status.
  • Pickup date filter is available.
  • View opens request detail panel.
AD-06 - Confirm request

As an operations admin, I want to confirm a request so that it becomes an operational booking.

Acceptance criteria
  • Request detail shows customer, trip, and vehicle preference.
  • Confirm action closes or updates the request detail flow.
  • The booking appears in the Bookings tab when stored in backend data.
AD-07 - Cancel request or booking

As an operations admin, I want to cancel with a reason so that cancellation history is traceable.

Acceptance criteria
  • Cancel opens a confirmation/reason modal.
  • Cancelled records stay visible in the Bookings tab.
  • Completed/cancelled bookings hide active assignment actions.
AD-08 - View booking details

As an operations admin, I want a booking detail panel so that I can inspect customer, trip, vehicle, driver, and timeline information.

Acceptance criteria
  • Panel shows Customer Information, Trip Information, Vehicle & Driver, and Booking Timeline.
  • Action block is hidden for Completed and Cancelled bookings.
AD-09 - Reassign resources

As an operations admin, I want to reassign vehicle or chauffeur so that operational changes can be handled quickly.

Acceptance criteria
  • Reassign vehicle modal lists available vehicles.
  • Reassign driver modal lists available chauffeurs.
  • Confirm closes the modal and should persist the new assignment in implementation.
AD-10 - Create manual booking

As an admin, I want to create a booking manually so that phone or concierge bookings can be entered into the system.

Acceptance criteria
  • Form captures customer, contact, pickup/drop, optional stops, vehicle type, date, time, and duration.
  • Additional stops can be added and removed.
  • Required fields validate before submit.

Business rules & developer notes

Implementation constraints and operational rules to preserve for the visible Admin.html portal flow.

Authenticated access only

All admin sections are internal. The auth screens hide the admin shell until login. Session expiry, logout, and permission checks should be enforced server-side, not only through hash routing or CSS display state.

Bookings are the default workspace

After login or password reset, admins land in Booking Management. This matches the Admin.html Login Securely and Reset & Open Dashboard actions.

Status changes must synchronize resources

Confirming, cancelling, or reassigning a booking should update the booking record and the visible vehicle/driver assignment fields consistently when connected to backend data.

Destructive actions need reasons

Cancel booking, disable vehicle, delete vehicle, suspend driver, and delete driver should require a reason and confirmation before the state change is committed.

Completed and cancelled trips are view-only

The HTML already hides booking action controls for completed or cancelled bookings. Production behavior should prevent reassignment, cancellation, or status progression after closure.

Drivers require valid operational status

Driver availability and license status should be checked before assignment. The driver table distinguishes Available, Assigned, On Leave, Unavailable, and On Trip states.

Fleet status controls assignment

Vehicle status should prevent invalid assignment. Available vehicles can be assigned; maintenance or disabled vehicles should be excluded from assignment lists once backend logic is connected.

Customer VIP state is editable

The Customers module supports VIP and Standard states from the row menu. Implementation should persist the change and keep filters/search in sync with the updated row state.

Communication settings are customer-facing

Notification Rules and Reminder Timing control how customers receive booking and trip updates. Test Notification should validate event and recipient details before sending.

Data shown is demo data

IDs, customer names, phone numbers, vehicles, drivers, dates, and revenue figures in the HTML are sample content. Replace with backend data and verify empty, loading, error, and permission-denied states.

This admin-flow file is a documentation artifact based on the supplied Admin.html structure. Confirm final API contracts, permissions, notification provider behavior, and persistence rules before implementation.
